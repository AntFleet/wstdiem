# wstdiem-loop-manager — Consolidated Audit Report

**Date:** 2026-06-03
**Scope:** entire codebase (`contracts/`, `src/`, `test/`)
**Method:** multi-agent audit — 12 specialist finders (security / code / architecture) across all subsystems, each finding re-checked by an independent adversarial verifier, then synthesized per dimension.
**Effort:** 90 agents, ~2.6M tokens.

## Executive Summary

The tool is **pre-production and simulation-only**: transaction broadcasting is intentionally disabled and `loop readiness` reports `broadcastAvailable: false` / `auditRequired: true`. **No Critical or High issue survived adversarial verification** — there is no live fund-loss exploit today because the high-risk on-chain path cannot yet broadcast.

The findings cluster around three themes that **must be closed before any broadcast path is enabled**:

1. **Fail-closed gates that are advisory rather than enforcing** — several checks the SPEC presents as hard gates are non-functional against a real RPC (notably the decoded-event cross-check in exit simulation) or rely on a single upstream guard with no defense-in-depth.
2. **Same-block / freshness provenance not consistently threaded** — exit simulation runs at `latest` while all evidence is pinned to the planning block (TOCTOU), and net-APY inputs aren't freshness-checked.
3. **A correctness bug that makes the flash-exit un-executable in the normal case** — the off-chain `repayAmountDiem` is snapshotted and repaid by assets while Morpho debt accrues every block, so a full-collateral withdraw reverts the atomic exit whenever any interest has accrued.

## Security Audit

The wstdiem-loop-manager is a pre-production, simulation-only operator tool with broadcasting intentionally disabled (`broadcastAvailable: false`, `auditRequired: true`), and the audit confirms no live fund-loss is exploitable today. The dominant theme is integrity of the fail-closed, evidence-gated design: several gates that the SPEC presents as enforcing are actually advisory or non-functional against a real RPC, and "same-block" provenance is asserted in some places but not consistently threaded through net-APY, simulation, and proof-artifact paths. The single most operationally likely correctness bug is an interest-accrual mismatch that renders the flash-exit un-executable. No Critical or High issues survived adversarial verification; the findings are latent defects and defense-in-depth gaps that must be closed before any broadcast path is enabled.

### [MEDIUM] Stale off-chain repay amount + full-collateral withdraw makes the flash-exit revert whenever interest has accrued
- **Location**: `contracts/LoopExecutor.sol:213-216` (computed off-chain in `src/loop/exitPlan.ts:71`)
- **Impact**: `repayAmountDiem` is a fixed asset amount snapshotted at the planning block and repaid with `shares=0`; Morpho Blue accrues borrow interest on every interaction, so actual debt > repayAmountDiem at execution. The asset-denominated repay leaves residual borrow shares, then `withdrawCollateral(maxWstDiemToSell)` (full collateral, `SPEC001.md:695`) reverts on the insolvent position, reverting the entire atomic exit. Fail-closed (no fund loss) but the exit is effectively un-executable for any non-zero borrow rate — the normal case — leaving the operator unable to deleverage during a depeg/liquidation scare.
- **Recommendation**: Repay by shares: read the owner's `position.borrowShares` on-chain inside the callback and call `IMorpho(morpho).repay(marketParams, 0, borrowShares, owner, "")` so the full debt clears regardless of accrual; or flash a small accrual buffer (`repayAmountDiem * (1 + eps)`) and refund DIEM dust via the existing path at `LoopExecutor.sol:229`. Add a foundry test where MockMorpho accrues interest between arm and callback.

### [MEDIUM] Decoded-event cross-check never runs in production, so the exit simulation passes-by-default without verifying the executor's real fee/repayment
- **Location**: `src/loop/simulator.ts:39-61, 63-100, 102-116, 233-270`; `src/loop/flashFeeProof.ts:84, 116-122`; `src/contracts/loopSimulationClient.ts:29-31`
- **Impact**: viem's `simulateContract` returns `{ result, request }` with no `.logs`, so `extractLogs` returns `[]`, `extractExitExecutionEvidence` returns `undefined`, and `exitEvidenceMismatch` short-circuits to `null`. The headline guarantee — cross-checking the decoded `LoopExitExecuted` event's `flashFee`/`totalFlashRepaymentDiem`/`repayAmountDiem` against the off-chain proof — only fires under the unit-test mock and is non-functional against a real RPC; absence of logs is treated as pass. Compounding this, the simulator never inspects `feeInclusiveRepayCovered`, `morphoRepayCovered`, `flashLoanLiquidityCovered`, or `flashFeeSource === 'unresolved'` to gate pass/fail, so those proof flags are decoration. The invariant holds today only because `exitPlan.ts:148` independently nulls params when `minDiemOut < totalFlashRepaymentDiem` — a single point of failure that any future param-builder bypasses with no signal.
- **Recommendation**: For `action === 'exit'`, do not treat `undefined` evidence as a pass — require a decoded `LoopExitExecuted` event and fail closed when absent, sourcing logs from a logs-bearing path (`publicClient.simulateCalls` or `debug_traceCall`) pinned to the evidence block. Additionally, make the simulator return `blocked`/`failed` before `passed` when `exitFlashFeeProof` is undefined, `flashFeeSource === 'unresolved'`, or any of `feeInclusiveRepayCovered`/`morphoRepayCovered`/`flashLoanLiquidityCovered` is not `true`, so the proof becomes an enforcing gate rather than relying solely on the upstream guard.

### [MEDIUM] Exit simulation runs at latest block while all evidence is pinned to the planning block (TOCTOU)
- **Location**: `src/loop/exitPlan.ts:41, 87-148, 171-188`; `src/loop/simulator.ts:226-261`; `src/contracts/loopSimulationClient.ts:29-34`
- **Impact**: `simulateContract`/`estimateContractGas` dispatch with no block parameter, executing against the node's latest block, which can be many blocks ahead of `planningBlock` (to which Morpho market/position, Curve quote, flash-pool balance, `repayAmountDiem`, and `minDiemOut` are pinned). No check ties the simulation block to the evidence block and no staleness bound exists on `planningBlock`. A simulation can pass while the proof's inputs are stale, so `minDiemOut >= repay + flashFee` may no longer hold at execution — weakening the same-block fail-closed invariant on the highest-risk path.
- **Recommendation**: Pass `blockNumber` equal to `planningBlock` to `simulateContract` and `estimateContractGas`, or re-read the block immediately before simulating and assert equality with every evidence block, failing closed on mismatch. Add a preflight check that route/liquidity/Morpho-debt/baseApy block numbers all equal the simulation block, and bound the maximum age of `planningBlock`.

### [LOW] `force` field is dead ABI with no on-chain effect
- **Location**: `contracts/LoopExecutor.sol:18, 153-185, 268-279`
- **Impact**: `LoopExitParams.force` is part of the committed ABI but is never read, never copied into `ExitFlashContext`, and never covered by `armedCallbackHash`. On-chain slippage protection is always `minDiemOut` (strictly safer than `SPEC001.md:764-766` implies), but the inert parameter could mislead a future CLI into mapping `--force` to a near-zero `minDiemOut` off-chain, exposing the owner to unbounded Curve slippage above the repayment floor with the contract silently accepting it.
- **Recommendation**: Remove the unused `force` field and document that on-chain slippage protection is always `minDiemOut`; or bind `force` into the armed context and assert that even under force `minDiemOut >= repayAmountDiem + flashFee` (the floor at `LoopExecutor.sol:263`), with an inline comment that force never disables the repayment-coverage invariant.

### [LOW] No on-chain Morpho `isAuthorized` check before borrow-on-behalf operations (SPEC §9 divergence)
- **Location**: `contracts/LoopExecutor.sol:214-216` (SPEC requirement at `SPEC001.md:731`)
- **Impact**: The executor performs `repay(onBehalf=owner)` and `withdrawCollateral(onBehalf=owner)` without first reading `morpho.isAuthorized(owner, address(this))`, diverging from the SPEC's stated invariant. Morpho Blue's own enforcement reverts if unauthorized (fail-closed, no fund loss), but the failure surfaces deep in the callback as an opaque Morpho error rather than an early, attributable executor revert, and the contract does not independently enforce the SPEC requirement.
- **Recommendation**: Add an explicit `IMorpho(morpho).isAuthorized(context.owner, address(this))` check at the top of the callback or in `_validateStaticContext`, reverting with a dedicated `ExecutorNotAuthorized` error; or, if intentionally delegating to Morpho, document the decision for the audit team.

### [LOW] Same-block freshness is not threaded through net-APY inputs (Morpho debt/market and baseApy blocks)
- **Location**: `src/loop/preflight.ts:381-401, 496-508, 518-522, 530, 544-549, 562-605, 819` (baseApy evidence from `src/loop/safetyEvidence.ts:24-42`)
- **Impact**: `readMorphoMarket`, the position read in `projectedPositionNotionalDiem`, `borrowRateView`, Curve balances, and `convertToAssets` are all called with no `blockNumber`, so they read chain head rather than the route-quote/planning block; only the flash-loan vs route-slippage equality is enforced (`preflight.ts:475`). Separately, `validateBaseApyEvidence` checks chainId/window/non-negative block but never bounds `baseApy.blockNumber` staleness against the route-quote block. Net-APY and curve-depth can therefore pass on a yield side and cost side measured at different blocks. (Exposure is reduced because `checkNetApy`/`checkCurveDepth` are skipped for `exit`, leaving rebalance/open — not yet implemented — and the secondary oracle gate.)
- **Recommendation**: Thread the route-quote `blockNumber` into `runLoopPreflight` and pin every read used by net-APY, curve-depth, oracle-deviation, and the Morpho debt/position reads to that block. Add a staleness bound that fails closed when `baseApy.blockNumber` is older than the route-quote block by more than a configured tolerance, and a gate asserting the Morpho-debt read block equals `routeSlippage.blockNumber`, mirroring `preflight.ts:475`.

### [LOW] Flash-loan / route same-block check is skipped when route evidence is absent
- **Location**: `src/loop/preflight.ts:475-477`
- **Impact**: The `blockNumber` equality assertion is nested inside `if (context.safetyEvidence?.routeSlippage !== undefined ...)`, so when `routeSlippage` is undefined the cross-block check is skipped and the flash-loan gate can pass on its own merits. Currently backstopped only because `checkRouteSlippage` independently fails on missing route evidence (`preflight.ts:686-704`, caught by `hasPreflightFailures` at `simulator.ts:205`). The flash-loan validator does not locally encode the dependency — fragile defense-in-depth that a future refactor making route-slippage optional would silently re-open.
- **Recommendation**: In `validateFlashLoanLiquidityEvidence`, require `routeSlippage` to be defined for exit and fail closed when missing, then assert `blockNumber` equality unconditionally rather than gating it behind the evidence's own presence.

### [LOW] Operator-facing flash-fee proof performs no same-block check between liquidity and route quote
- **Location**: `src/loop/flashFeeProof.ts:89-122`
- **Impact**: `buildExitFlashFeeProof` derives `matchingLiquidity` from provider/factory/pool/loanToken/requestedLoan/valid/availableLoan and sets `feeInclusiveRepayCovered` from `minDiemOut >= totalFlashRepaymentDiem`, recording `flashLoanLiquidityBlockNumber` but never comparing it to the route-quote or Morpho-debt block (the route block is not even an input). The blocking decision is safe (preflight enforces the cross-block check and the projection path shows `blocked`), but in the live simulator path the operator-readable artifact can report `feeInclusiveRepayCovered: true` from liquidity at a block unrelated to the route quote, contradicting `SPEC001.md:741`.
- **Recommendation**: Pass the route-quote (and Morpho-debt) block into `buildExitFlashFeeProof` and require `flashLoanLiquidity.blockNumber` to equal them; when they differ, emit `feeInclusiveRepayCovered: 'blocked'` with a stale-evidence reason, matching the SPEC's same-block requirement for the printed decision.

### [LOW] Deadline is never validated as in-the-future by any off-chain gate (SPEC §5 non-skippable property)
- **Location**: `src/loop/preflight.ts:719-824` (SPEC at `SPEC001.md:731, 765`; construction at `params.ts:60-62`, `exitPlan.ts:171-173`)
- **Impact**: `SPEC §5` lists `deadline` as a property `--force` must never skip, yet neither `runLoopPreflight`, `buildLoopReadiness`, nor `simulateLoopExecutorCall` validates `params.deadline` against now/chain timestamp. Normal construction yields a fresh deadline, but a caller-supplied stale `nowSeconds` (`cli/loop.ts:55`) or replayed calldata could produce an already-elapsed deadline that all off-chain gates accept, relying entirely on the unaudited executor to revert. Defense-in-depth gap, not a direct exploit.
- **Recommendation**: Add a preflight gate that fails closed when `params.deadline <= currentBlockTimestamp` (or `<= nowSeconds + minimum margin`) for both exit and rebalance, enforcing the deadline independently of the executor and of `--force`.

### [LOW] `interpolateEnv` silently drops missing env vars, producing malformed-but-valid URLs
- **Location**: `src/config/load.ts:165-170`
- **Impact**: A missing `${VAR}` is replaced with `""` and only returns `null` when the entire string is empty after trim, so partially-interpolated URLs (e.g. `https://eth.example.com/v3/` or `https://.infura.io/v3/key`) pass `z.string().url()` validation on RPC and webhook URLs. An RPC endpoint whose API key was silently dropped — the source of financial truth for the fail-closed gates — or a webhook with a dropped auth path segment is accepted without error. Operator footgun, not attacker-exploitable.
- **Recommendation**: Track undefined referenced vars during interpolation and throw (e.g. `Unresolved env var(s) in config value: ...`) so a silent malformed URL becomes an explicit fail-closed startup error.

### [LOW] RPC finalized block number/timestamp default to 0 with no positivity bound
- **Location**: `src/contracts/rpc.ts:47-52`
- **Impact**: `selectBestRpcEndpoint` coerces `block.number ?? 0n` and `Number(block.timestamp ?? 0)`, so a misbehaving/single RPC returning a null finalized block propagates `blockNumber: 0` / `blockTimestamp: 0` as a successful selection rather than a failed read. Downstream `safetyEvidence.ts:31` only rejects `< 0n`, so 0 passes; it is largely self-correcting (chainId mismatch fails freshness, a 0-timestamp trips the `rpc_stale` WARN at `evaluate.ts:143`), but using a zero default for block-height/freshness truth is fail-open-shaped.
- **Recommendation**: Treat a missing/zero finalized block as a failed read — throw inside the try block when `block.number` is null/undefined/`0n` (and require a positive timestamp) so the endpoint enters `failures[]` and forces fail-over or a hard error instead of propagating block 0 as canonical state.

### [LOW] Telegram alert config is declared but never delivered (misleading dead config)
- **Location**: `src/alerts/deliver.ts:62-86` (declared at `src/config/load.ts:83-86`, `src/config/defaults.ts:59-62`, `src/types/domain.ts:71-74`, `config.example.yaml`)
- **Impact**: `deliverConfiguredAlerts` — the only delivery path — implements stderr and webhook channels only; no code reads `botTokenEnv` or calls `api.telegram.org`. An operator who configures Telegram reasonably expects CRITICAL alerts (HF<1.4, oracle deviation, spread compression) to reach it, but they are silently confined to stderr/webhooks; for a leverage monitor this can mean a missed liquidation-avoidance window. (No secret-leak surface, since no token is read or transmitted.)
- **Recommendation**: Either implement `sendTelegram` (read the token strictly from `process.env[botTokenEnv]`, never log/store it, reuse the `AbortSignal.timeout` pattern, and add a `telegram:delivered/failed` entry to `delivered[]`), or remove the telegram block from schema/defaults/example until implemented. Add a test asserting `deliveredChannels` reflects all configured channels.

### [LOW] `exitEvidenceMismatch` omits `minDiemOut`, `owner`, and `wstDiemSold` from the executor-event reconciliation
- **Location**: `src/loop/simulator.ts:102-116`
- **Impact**: When an event is decoded, the mismatch check compares only `flashFee`, `totalFlashRepaymentDiem`, and `repayAmountDiem` — never `event.owner` vs params owner, `event.diemReceived` vs off-chain `minDiemOut`, or `event.wstDiemSold` vs `maxWstDiemToSell`. The protective floor `minDiemOut` is not reconciled at all, so an event matching on fees but corresponding to a different owner or a `diemReceived` below `minDiemOut` would not be caught here. Latent given the cross-check itself does not run in production (above), but this is where the `minDiemOut` reconciliation belongs.
- **Recommendation**: Extend `exitEvidenceMismatch` to assert `evidence.owner === params.owner`, `evidence.diemReceived >= params.minDiemOut`, `evidence.wstDiemSold <= params.maxWstDiemToSell`, and `evidence.totalFlashRepaymentDiem <= proof.minDiemOut`, failing closed on any divergence.

### [LOW] `deriveConfiguredExitFlashFee` does not validate feeTier range or pool/pair pairing
- **Location**: `src/loop/flashFeeProof.ts:19-54`
- **Impact**: It validates provider/factory/pool/loanToken/pairToken/feeTier non-null and `loanToken == DIEM`, but not that `feeTier` is a sane Uniswap V3 tier, that `pairToken != loanToken`, or that the pool equals `factory.getPool(loanToken, pairToken, feeTier)` — those structural checks live only in `preflight.checkExecutorFlashConfig` (`preflight.ts:206-257`) and the on-chain constructor (`LoopExecutor.sol:281-290`), which run only when a client is present. The projection-only path (`cli/loop.ts:219`) builds the proof without a client, so a mis-set feeTier (e.g. `100000` = 10%) yields a plausible-but-wrong, internally consistent fee. (Low: the binding pass path runs preflight and the on-chain `FeeMismatch` at `LoopExecutor.sol:207` is the final guard; the projection path is also unconditionally broadcast-blocked.)
- **Recommendation**: Validate `feeTier` against the allowed Uniswap V3 set and assert `pairToken != loanToken` inside `deriveConfiguredExitFlashFee`, and mark the proof reason explicitly when structural pool/factory validation has not been performed (no client).

### [INFO] Strict fee equality and theoretical `amount * feeTier` overflow in `_expectedFlashFee`
- **Location**: `contracts/LoopExecutor.sol:207-208, 318-320`
- **Impact**: No practical exploit. `_expectedFlashFee` multiplies `amount * feeTier` before `ceilDiv`, which only overflows uint256 above ~1.16e73 for the 1% pool — infeasible for an 18-decimal token. The callback's strict `flashFee == _expectedFlashFee(...)` and `otherFee == 0` are desirable fail-closed properties (any fee surprise reverts), and the constructor's canonical-pool pin guarantees `feeTier == pool.fee()`. Documented for explicit confirmation of the no-overflow and mulDivRoundingUp-vs-ceilDiv parity assumptions.
- **Recommendation**: Optionally use OpenZeppelin `Math.mulDiv`/a `mulDivRoundingUp` helper to make the parity self-evident and overflow-proof, and add an inline comment that `feeTier == pool.fee()` is guaranteed by the constructor. No functional change required.

### [INFO] Off-chain and on-chain `ceilDiv` use different formulas (latent fee-parity drift)
- **Location**: `src/loop/flashFeeProof.ts:15-17, 45`; `contracts/LoopExecutor.sol:318-320, 331-333`; `src/loop/preflight.ts:198-204`; `src/loop/readiness.ts:63-69`
- **Impact**: `flashFeeProof.ts` uses `(numerator + denominator - 1)/denominator`, while the on-chain `_ceilDiv`, `preflight`, and `readiness` use `numerator == 0 ? 0 : ((numerator-1)/denominator)+1`. These are mathematically identical across the real domain (`repayAmountDiem > 0`), and `preflight.ts:243` asserts runtime parity against the executor — but three copies in two forms (the proof copy being the non-canonical one) invite future drift that would silently break the `minDiemOut >= repay + flashFee` parity.
- **Recommendation**: Centralize a single `ceilDiv`/`expectedFlashFee` helper shared by `flashFeeProof`, `preflight`, and `readiness`, mirroring the on-chain `numerator == 0` form exactly, and unit-test parity against `LoopExecutor.sol._expectedFlashFee` across boundary amounts.

### [INFO] `projectedHealthFactor` gate uses `>=` against 1.7 while SPEC §5 specifies strict `>`
- **Location**: `src/loop/preflight.ts:354-361` (SPEC at `SPEC001.md:529`; default at `defaults.ts:46`, validated at `load.ts:70`)
- **Impact**: `checkProjectedHealthFactor` compares `projected >= minimum` (1.7), so a projected HF of exactly 1.7 passes where SPEC's strict inequality would block. Diverges only at the exact boundary value; negligible in practice. All other comparators match SPEC.
- **Recommendation**: If strict SPEC fidelity is desired, change the comparison to `projected > minimum` (and the message accordingly) so exactly 1.7 fails closed.

### [INFO] `priceImpactBps` floors sub-PPM Curve impacts to 0
- **Location**: `src/loop/routeQuote.ts:44-50, 107-108, 132`
- **Impact**: `Number((diff * 1_000_000n) / expectedOut) / 100` floors any impact below 1 PPM (0.0001%, not 0.01%) to 0 bps, so a quote marginally worse than NAV at noise-level magnitude can read as zero impact and pass the secondary `maxCurvePriceImpactBps` cap. The binding protections — the bigint `minDiemOut` slippage floor and fee-inclusive repay coverage — are unaffected; this is an advisory metric.
- **Recommendation**: Round the impact up (`ceilDiv(diff * 1_000_000n, expectedOut)`) before converting, and/or keep the comparison in bigint; document that `priceImpactBps` is advisory and `minDiemOut` is the binding floor.

### [INFO] Malicious config `__proto__` key hijacks the merged object's prototype before zod re-parse (fully mitigated)
- **Location**: `src/config/load.ts:172-213`
- **Impact**: `deepMerge` iterates `Object.entries(patch)` and executes `output["__proto__"] = value`, reassigning the local merged object's prototype to attacker-supplied content. However there is no process-wide `Object.prototype` pollution, and `configSchema.parse()` returns a fresh clean-prototype object containing only schema-declared keys, so the hijacked prototype never reaches the consumed `AppConfig`. Latent only if future code consumes the pre-zod `merged` object directly or switches to `.passthrough()`/`.catchall()`.
- **Recommendation**: Defense-in-depth: drop `__proto__`/`constructor`/`prototype` keys from parsed YAML, build merge outputs on null-prototype objects (`Object.assign(Object.create(null), base)`), and skip those keys in `resolveEnvPlaceholders`, so the loader stays fail-closed regardless of downstream evolution.

### [INFO] Missing test coverage for interest-accrual revert, stale repay, and authorization failure
- **Location**: `test/foundry/LoopExecutor.t.sol:65-80, 461-487`
- **Impact**: `MockMorpho` repays exactly the requested assets and withdraws exactly the requested collateral with no interest accrual, no health/residual-debt check, and no `onBehalf` authorization check, so the happy-path test cannot surface the interest-accrual exit revert (the medium finding above). There is no test where `repayAmountDiem < live debt`, none asserting a clean revert when Morpho authorization is absent, and the fork tests gate readiness on env but never invoke `exit()` against live accruing debt. The most operationally likely failure mode and the SPEC-mandated authorization invariant are unverified by CI.
- **Recommendation**: Add a `MockMorpho` variant that (a) accrues interest between arm and callback and asserts a successful repay-by-shares unwind or a deliberate attributed revert; (b) reverts `withdrawCollateral` when residual debt remains; (c) reverts when the executor is not authorized. When live liquidity and a funded owner exist, extend the fork readiness test to actually invoke `exit()` and assert no retained balances and full debt clearance.

**Strengths**: The fail-closed, evidence-gated architecture is genuinely well-built where it is wired correctly. The constructor pins the configured Uniswap V3 pool to the factory-derived canonical pool, guaranteeing `feeTier == pool.fee()` and making the strict fee-equality and `otherFee == 0` checks robust fail-closed guards. The on-chain `minDiemOut` slippage floor is enforced unconditionally regardless of the (inert) `force` flag, and the fee-inclusive `minDiemOut >= repayAmountDiem + flashFee` invariant is asserted both off-chain and on-chain. `exitPlan.ts` correctly pins its own Morpho market/position/quote/liquidity reads to a single planning block, the bigint slippage floor for `minDiemOut` is precise and correctly protective, the EVM-atomic exit means even the interest-accrual bug loses no funds, and broadcasting is hard-disabled with `auditRequired: true` — the appropriate posture for a pre-production, audit-pending leverage tool.

## Code Quality Audit

The codebase exhibits a generally strong, fail-closed engineering posture: evidence-gating, block-pinned route/flash-loan reads, and structured readiness reporting are wired throughout, and most on-chain reads degrade to structured failures. The confirmed findings are dominated by latent and pre-production-only issues rather than active fund-loss bugs, which is consistent with the simulation-only, audit-pending status. The most material gaps are an unimplemented SPEC-mandated alert channel, a serialization crash on the headline live-simulation success path, and untested core contract safety guards. None of the confirmed findings represent an exploitable fund-loss path in the current build, but several would bite at deployment if left unaddressed.

### [MEDIUM] Telegram alert delivery is configured but never implemented
- **Location**: `src/alerts/deliver.ts:50-86` (config surface at `src/config/load.ts:81-87`, `src/config/defaults.ts:57-63`, `src/types/domain.ts:69-75`)
- **Impact**: SPEC §3 (SPEC001.md:375) mandates Telegram as a delivery channel. `deliverConfiguredAlerts` only delivers to stderr and `webhookUrls` — a repo-wide grep finds zero Telegram delivery code. Operators who configure a bot token + chat id silently receive no Telegram alerts (including CRITICAL HF/oracle-deviation alerts), and the gap is invisible in the `alerts test` delivered-channels report.
- **Recommendation**: Implement `sendTelegram(botToken, chatId, payload)` via undici POSTing to `https://api.telegram.org/bot<token>/sendMessage` with Markdown-safe text, wired into `deliverConfiguredAlerts` when `telegram.chatId !== null` and the bot-token env var is set; push a `telegram:delivered`/`telegram:failed` label like the webhook path. Until then, surface a startup/readiness warning that Telegram is configured but unsupported so the gap is not silent.

### [MEDIUM] `loop simulate --live` (non-JSON) crashes serializing raw bigints in execution evidence
- **Location**: `src/cli/index.ts:50` (evidence source at `src/loop/simulator.ts:84-94`, `239-247`, `262-270`)
- **Impact**: On a passing exit live-sim without `--json`, `runAction` reaches `console.log(JSON.stringify(data, null, 2))` — the plain stringifier, not the bigint-safe `stringifyJson` from output.ts. The returned `exitExecutionEvidence` (and other `LoopSafetyEvidence`/params) carry raw bigints, so `JSON.stringify` throws `TypeError: Do not know how to serialize a BigInt`, which is caught and re-wrapped as `UNEXPECTED_ERROR` — masking an otherwise-successful run of the headline safety tool in human-readable mode.
- **Recommendation**: Replace `console.log(JSON.stringify(data, null, 2))` at `index.ts:50` with `console.log(stringifyJson(data))` (already exported from output.ts) so both `--json` and plain modes use the same bigint-aware replacer.

### [MEDIUM] Fork tests no-op without `BASE_RPC_URL`; deploy/liquidity/unwind proofs never run in default CI

## Architecture Audit

The architecture is a clean, layered TypeScript operator CLI (cli -> loop/metrics -> contracts -> abi -> types) wrapped around an exit-only Solidity executor, and its fail-closed, evidence-gated intent is genuinely pervasive throughout the codebase. - **Impact**: The pervasive fail-closed philosophy is expressed through four structurally different systems with three result-envelope shapes: `PreflightCheck[]` + `hasPreflightFailures()`; `projectLoopCommand`'s free-form `string[]` blockers plus a separate `safeToProceed/blocked` pair derived from `blockers.length` (not from the structured checks); `buildLoopReadiness`'s near-duplicate `ReadinessCheck[]` plus its own blocker strings; and `buildLiveLoopExitPlan` signalling failure via `params: null`. `PreflightCheck` and `ReadinessCheck` are duplicate types. Because the CLI projection's `blocked/safeToProceed` derive from a string array rather than the structured preflight verdict, the two can drift — some gates (e.g. `loopExecutor === null`) appear only in `blockers`. A new safety gate must be added in up to four places and worded consistently. This is contained today only because the audit gate unconditionally blocks broadcast; it becomes a latent correctness bug once live execution is enabled.
- **Recommendation**: Collapse `PreflightCheck` and `ReadinessCheck` into one `Gate` type in `loop/types.ts` and add one reducer `Gate[] -> {status, blockers}`. Have `projectLoopCommand`, `readiness`, `preflight`, and `exitPlan` all emit `Gate[]` and derive `blocked`/`safeToProceed`/`readiness` from that single reducer, giving one auditable definition of "allowed to proceed" and making a new gate a single-site change.

### [LOW] Exit force-sells 100% of collateral on Curve with only minDiemOut as the floor; no partial-unwind path

- **Location**: `contracts/LoopExecutor.sol:216,218-220,229`; `src/loop/exitPlan.ts:179`
- **Impact**: After repaying, the callback withdraws exactly `maxWstDiemToSell` and sells the full amount on Curve, and the CLI always sets `maxWstDiemToSell` to the owner's entire collateral. There is no on-chain partial-unwind/rebalance path: any collateral beyond what `totalRepayment` needs is converted to DIEM and refunded as dust rather than re-supplied as collateral. Selling 100% maximizes Curve price impact — exactly what the SPEC's off-chain depth gates try to bound — and with `--force` the off-chain slippage guard is skipped, leaving only `minDiemOut >= repay + fee`, which still permits large unbounded slippage above that floor. Correct for full exits (SPEC documents this as the full atomic unwind slice) and no funds are lost, but a real position owner may realize materially less value than a debt-sized partial sale.
- **Recommendation**: For the production design, separate "repay amount" from "collateral to sell": sell only enough wstDIEM to cover `totalRepayment` plus a buffer and return remaining collateral as wstDIEM. If full-liquidation-only is intended, document it explicitly and have the CLI/readiness surface the projected price impact of selling 100% of collateral, and ensure the planner's `minDiemOut` embeds the curve price-impact guard so `--force` is the only bypass.

### [LOW] exit() returns an all-zero LoopResult and performs no on-chain post-state assertion

- **Location**: `contracts/LoopExecutor.sol:234,183-184,115-124,220`
- **Impact**: The `LoopExitExecuted` event adequately supports the off-chain proof cross-check, but `exit()`'s return value is hard-coded to `LoopResult{0,0,0}` even though the SPEC-defined interface carries `collateralWstDiem`/`borrowedDiem`/`healthFactorWad` — any caller relying on the documented return shape gets meaningless data. The executor never reads back Morpho position state, so it makes no on-chain assertion (e.g. `borrowShares == 0` for a full exit); correctness rests entirely on Morpho's own reverts plus the atomic flash-repayment balance check, which is sound for exit but provides no explicit post-state safety check. Additionally, `wstDiemSold` in the event emits the input `maxWstDiemToSell` (the Curve `exchange` return is discarded), conflating "requested to sell" with "sold". Note the SPEC's post-state HF verification is specified for the open sequence, not exit.
- **Recommendation**: Prefer (a): read Morpho `position(marketId, owner)` at the end of the callback, populate `LoopResult`, and assert post-exit invariants (full exit -> `borrowShares == 0`) for a real on-chain safety check. Otherwise (b) remove the return value from `exit()` or loudly document that it is always zero for the exit slice so off-chain code does not depend on it.

### [INFO] Client-port interfaces live in consumer modules, inverting the cli->loop->contracts dependency direction

- **Location**: `src/contracts/loopSimulationClient.ts:4`; `src/contracts/routeQuoteClient.ts:4`; `src/contracts/backfillClient.ts:4` (interfaces at `loop/simulator.ts:21`, `loop/routeQuote.ts:11`, `metrics/backfill.ts:29`)

### [INFO] buildLiveLoopExitPlan concentrates orchestration, reads, fee math, evidence, and gating in one 170-line function

- **Location**: `src/loop/exitPlan.ts:20-190` (esp. block coupling at 113 vs 41/130)
- **Impact**: This is the de-facto live-exit orchestrator and the single most complex node in the loop package, mixing data acquisition with policy across five bespoke early-return blocks. The same-block invariant is implicit: the pool balance is read at `quoteResult.evidence.blockNumber` while everything else uses `planningBlock`; they are equal only because the quote was requested with `blockNumber: planningBlock`. Correct today, double-guarded by `preflight.ts:475`, but a likely growth point for a future open/rebalance planner.
- **Recommendation**: Split into a pure reader (market/position/borrowedDiem at a fixed block), the existing quote call, a pure `assembleExitFlashLoanEvidence`, and one `Gate[]` evaluator (per the gating consolidation above). Pass `planningBlock` explicitly into the pool-balance read so the same-block invariant is visible at the call site.

### [INFO] Executor implements only exit(); open()/rebalance()/AutoDeleverageExecutor from the SPEC interface are unimplemented

- **Location**: `contracts/LoopExecutor.sol:64,153`; `src/loop/params.ts:31-90`
- **Impact**: By design for this slice (SPEC OQ2/OQ4, broadcast disabled). `open()`/`rebalance()` are absent from the contract and `buildLoopOpenParams`/`buildLoopExitParams` return null fail-closed; the open/rebalance surfaces (flash + vault.deposit + borrow-on-behalf; keeper trust) remain entirely unwritten and unaudited. The accepted `force` field is correctly never consulted on-chain. Note: `buildLoopRebalanceParams` actually returns a populated object, but since the contract has no `rebalance()` it would revert on-chain — still blocked, via a different mechanism.
- **Recommendation**: Keep exit-only. Before building open()/rebalance(), write a dedicated Solidity spec for the borrow-on-behalf and vault.deposit flows and a separate trust-model review for any keeper variant. Document in NatSpec that this is the exit-only slice so the unimplemented SPEC ABI methods are not mistaken for missing.

### [INFO] Trust model concentrates power in the owner EOA plus a standing, unscoped Morpho authorization the executor cannot revoke

- **Location**: `contracts/LoopExecutor.sol:214,216,253`
- **Impact**: `exit()` is correctly owner-gated and the full repay->withdraw->swap->flash-repay->refund sequence is atomic inside a single callback. The latent concern is the standing `morpho.setAuthorization` capability: once granted it is not scoped to a single intent and would persist for any future exit (or a future open/borrow under the same authorization), widening blast radius if the executor key/logic were compromised. There is no pause/admin (good for trust-minimization, but a logic bug can only be abandoned via redeploy + re-authorize).
- **Recommendation**: Document that operators should authorize the executor only immediately before an exit and revoke afterward (or add a `loop revoke-authorization` command). When designing open()/rebalance(), require an explicit owner-signed, deadline+nonce-bound (ideally EIP-712) intent for any borrow-on-behalf so a future keeper cannot act outside a signed scope. No change to the exit slice itself.

### [INFO] Reliance on external getPool/Morpho/Curve is fail-closed on the happy path, but oracle/HF and Curve depth are off-chain-only and marketParams oracle/irm/lltv are unvalidated

- **Location**: `contracts/LoopExecutor.sol:196,214-216,220,222-224,274-275,326-329,287` (off-chain gates in `src/loop/preflight.ts`)
- **Impact**: External-call failures are handled defensively: `getPool` mismatch reverts at construction and in-flow; Morpho calls use `onBehalf=owner` and revert on bad auth/amount; Curve is floored by `minDiemOut` three ways (minDy arg, balance-delta, totalRepayment); `_callOptionalReturn` handles non-standard ERC20s. The canonical-pool check correctly uses live `factory.getPool` equality (a valid "or equivalent" per SPEC, more robust against init-code-hash mistakes than CREATE2, at the cost of repeated STATICCALLs per exit). The residual architectural risk: oracle-deviation and Curve-depth gates live ONLY off-chain, so the planner is in the trusted computing base for anything beyond fee-inclusive repayment; and `exit()` validates only `loanToken`/`collateralToken` against config, not `oracle`/`irm`/`lltv`, so a wrong-but-superficially-matching market could be targeted. Acceptable now (broadcast disabled, owner-initiated) but latent.
- **Recommendation**: Validate full `marketParams` (pin expected `oracle`/`irm`/`lltv==0.86e18` as immutable config and assert, or compute and require the Morpho market id). Document that oracle-deviation and Curve-depth gates are off-chain-only (planner in the trust boundary); consider an on-chain oracle sanity read before the swap for defense-in-depth. Optionally cache the canonical pool in an immutable to remove repeated registry reads, and flatten `flashConfig`/`protocolConfig` into individual `immutable` fields to make the pinned-config fail-closed property compiler-enforced and remove per-exit SLOADs (update the fork test's tuple destructuring accordingly). Keep the redundant `minDiemOut` checks.

**Strengths.** The executor's exit path is genuinely well-built for a first target: atomicity is correctly guaranteed by running the entire unwind inside a single `uniswapV3FlashCallback`, the canonical-pool validation via live `factory.getPool` equality is sound and arguably safer than hand-rolled CREATE2, the Curve fill is protected by a triple belt-and-suspenders (`minDy` + balance-delta + totalRepayment floor), non-standard ERC20s are handled via `_callOptionalReturn`, the constructor is fail-closed, and `exit()` is strictly owner-gated. Off-chain, the fail-closed, evidence-gated philosophy is real and pervasive, the unimplemented open/rebalance builders correctly return null rather than emitting unprotected calldata, and the `--force` flag never weakens the on-chain `minDiemOut`/fee gate. The findings above are overwhelmingly latent maintainability and pre-deployment hardening items, not active defects.
